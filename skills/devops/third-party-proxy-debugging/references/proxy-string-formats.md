# Proxy String Formats — Multi-Vendor Parser Reference

When a user buys from 2+ proxy vendors, they paste lists in **completely different formats**, often mixed in the same paste. A reusable parser must auto-detect all of these, not require the user to pick a format.

## Formats seen in the wild

| Format | Example | Source |
|---|---|---|
| `host:port:user:pass` (4 colons) | `us.cliproxy.io:3010:zmeq107638-region-JP-sid-X:f57bgpcg` | cliproxy, 711proxy, most residential vendors |
| `socks5://user:pass@host:port` | `socks5://user:pass@1.2.3.4:1080` | Standard URL form, generated by some dashboards |
| `socks5h://user:pass@host:port` | `socks5h://u:p@h:1080` | Same as above but force remote DNS |
| `http://user:pass@host:port` | `http://myuser:mypass@proxy.example.com:8080` | HTTP proxies, less common for residential |
| `user:pass@host:port` (no scheme) | `user:pass@1.2.3.4:1080` | Shorthand, default to SOCKS5 |
| `host:port@user:pass` (reversed!) | `1.2.3.4:1080@user:pass` | Some vendor dashboards, surprisingly common |
| `host:port` (no auth) | `1.2.3.4:1080` | IP-whitelisted endpoints |
| `socks5://host:port` (URL no auth) | `socks5://us.cliproxy.io:3010` | Whitelisted SOCKS5 |
| `user:pass:host:port` (4 colons, reversed) | `myuser:mypass:1.2.3.4:1080` | Rare but exists |

## Detection algorithm

The trick is to NOT require the user to choose. Detect per-line:

```
1. Strip whitespace + inline comments (# or //).
2. Match URL prefix /^(socks5|socks5h|socks4|http|https):\/\/(.+)$/i:
   - If yes, parse the rest as either user:pass@host:port OR host:port (no auth).
3. If contains '@':
   - Split on '@' once.
   - Test left=user:pass + right=host:port, then reverse.
   - Whichever side has a valid (host-like, port-number) pair is the host side.
4. Else split on ':':
   - 2 parts → host:port (no auth)
   - 4 parts → try host:port:user:pass first (most common),
              fall back to user:pass:host:port if first segment isn't host-like
   - 5+ parts → password may contain ':'. Try host:port at front or back, treat middle as user:pass.
```

## "Host-like" heuristic

To disambiguate `1.2.3.4:1080:user:pass` from `user:pass:1.2.3.4:1080`, classify each segment:

- **Port-like**: matches `/^\d+$/`, value 1-65535
- **Host-like**: localhost, OR IPv4 regex `/^\d{1,3}(\.\d{1,3}){3}$/`, OR domain regex `/^[a-zA-Z0-9][-a-zA-Z0-9.]*\.[a-zA-Z]{2,}$/`
- **Neither**: probably user or password

If segments 1+2 are host+port → it's `host:port:user:pass`. If segments 3+4 are → it's `user:pass:host:port`.

## URL-encode the credentials when building proxy URLs back

Vendor usernames frequently contain `-`, sometimes `_`, occasionally `.`. Passwords have been observed with `@`, `:`, `/`. ALWAYS `encodeURIComponent` user and password before reassembling:

```js
function buildProxyUrl(p) {
  const proto = p.protocol || "socks5";
  const auth = p.username
    ? `${encodeURIComponent(p.username)}:${encodeURIComponent(p.password || "")}@`
    : "";
  return `${proto}://${auth}${p.host}:${p.port}`;
}
```

Forgetting this is the #1 cause of "curl works but my Node code fails" — Node's URL parser will silently truncate at the unescaped `@` in the password.

## Dedup signature

For batch import where the user pastes the same proxy twice across vendors, dedup by:

```js
function proxySignature(p) {
  return `${p.protocol || "socks5"}://${p.host}:${p.port}|${p.username || ""}`;
}
```

Note: do NOT include password in the signature — same proxy with two passwords means user has two accounts, treat as two entries. Including the password in the signature would let a stale password create a duplicate entry on re-import.

## Vendor auto-detect

Useful for UI display ("which vendor is this from?"). Pattern match on hostname:

| Pattern in host | Vendor |
|---|---|
| `cliproxy` | cliproxy |
| `711proxy` or `rotgb` | 711proxy |
| `smartproxy` | smartproxy |
| `brightdata` or `luminati` | brightdata |
| `oxylabs` | oxylabs |
| `iproyal` | iproyal |
| `packetstream` | packetstream |
| (anything else) | `unknown` |

Showing vendor in the IP-pool table helps users mentally group entries when troubleshooting per-vendor whitelist issues.
